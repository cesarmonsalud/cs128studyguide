Tools

  • git
    • git clone
      • clone a git repo from a webhost or filepath
    • git init
      • initializes a repository with the necessary hidden files.
    • git add
      • adds a specified file to the staging area
      • using -u will update all modified files to the staging area
      • using -A will add all files (modified and new) to the staging area
    • git commit
      • commits all files currently in the staging area to the active branch
      • using -m will automatically pass a comment for that commit
    • git push
      • pushes your active repo to another branch
      • i.e. git push origin master will push your work to the original repo master branch.
    • git pull
      • pulls any new updates from the origin repository
      • will try to merge with your working repository
  • make
    • makefiles are out of scope, just understand that calling make <target> will run the specified actions detailed in the makefile.
    • standard targets are make file make test and make clean
  • clang
    • runs a standard c++ compiler.
    • clang <main.cpp> -o <output_executable> is the simplest version of the command you can run.
    • you can look at additional flags on the clang website
  • VSCode
    • run code . to open vscode on a working directory
    • know your general navigation techniques
    • set up c++ development for vscode here
  • GDB
    • run gdb on a program using gdb --args ./exe [args] (omit --args if there are no arguments)
    • r for run, start execution
    • break file:lineno to set a breakpoint
    • bt to readout the backtrace
    • n for next line (aka "step over")
    • s for step into
    • finish continue debugging after function returns
    • c continues execution until next breakpoint
    • quick reference and gdb-peda, a nicer wrapper for gdb
  • Valgrind
    • you need two things, the rest is good luck
    • valgrind ./yourprogram if that doesn't work,
    • valgrind --leak-check=full ./yourprogram
    • read the stack traces to find what is calling your allocations and frees
  • Clang AddressSanitizer
    • compiler based address sanitizer (like valgrind)
    • helps identify pointer faults
    • instructions available here
    • simplest pass is clang -g -fsanitize=address file.cpp -o executable
  • Clang MemorySanitizer
    • compiler based memory sanitizer (like valgrind)
    • helps find memory read errors
    • instructions available here
    • simplest pass is clang -g -fsanitize=memory file.cpp -o executable
  • catch2
    • helps run and manage tests
    • include the catch.hpp and necessary headers with code to test

  • Know your OOP principles
    • Creating a class class square{};
    • Creating an object Square a(5)
    • Know what to make private and public (encapsulation)
      • private stuff: variables, internal functions, cleaners
      • public stuff: modifier functions, structors, getters, setters
      • protected stuff: your own discretion, just know that all derived classes can use it
    • deriving a class: class rectangle : public square {}
      • colon signifies inheriting
      • access mode tells what to change access rights from square to rectangle
        • public: public stays public, protected stays protected
        • protected: public and protected are protected
        • private: public and protected are private (you'll never use these two 99% of the time)
  • Memory and Memory Management
    • Know your process layout
      • Environment Variables (you never have to worry about this)
      • Stack (grows down with functions, self cleans as functions returns)
      • Heap (grows up with allocations, reogranizes and shrinks with frees)
      • Data (various global variables)
      • Text (various global text variables)
    • new and delete
      • new allocates enough space for whatever you tell it to, and returns a pointer
      • delete takes a pointer and frees that space back up
      • making a C array with new requires you to delete it with delete[]
        • e.g. int * ptr = new int[15] must have delete[] ptr
    • Rule of 0/3/5
      • Rule of 0
        • Don't make classes if you don't have to
        • Don't make other classes have custom -tors for other classes that have them, let the detors do their work.
      • Rule of 3
        • If you need a destructor, then you need a copy constructor and an assignment constructor (works for any of these 3)
          • Allocated memory means you need to specify a destructor
          • Which means you need to specify how much memory to allocate and where to get the copied data from the original object when copying.
          • assignment constructor follows suit with copying and reassigning values.
          • Bottom line: Allocating memory requires all three methods to be implemented
      • Rule of 5
        • This is about adding move constructors when you've done rule of 3.
        • Not necessary, but know how to make move tors.

    • Memory leaks
      • you forgot to free your pointers, this means that if you allocate memory on a loop, you'll eventually crash your heap.
    • Memory errors
      • poor handling of pointers, allocation, and freeing
    • Dangling pointers
      • freeing a pointer makes it point to memory you don't own
      • also, moving a pointer to memory don't own is technically dangling
      • make sure to NULL dangling pointers, and then also go through the pain of null checking your pointers.

  • Data Structures and Traversals
    • Vectors
      • It's basically an array with some nice wrapping and mangement
      • Standard [] operator
      • push_back pop_back
      • iterators
      • memory, size, and capacity mangement
    • Lists
      • Like a vector, but more freedom to insert and remove whereever
      • Linked list implementation, doubly linked list allows dual traversal
    • Binary Trees
      • Starts at the root node
      • Each node has two pointers to other nodes
      • To make it a Binary Search Tree, you put items that are lower than the current node on the left, and items that are larger on the right. Note that this will construct specific trees and it's easy to tell invalid search paths
    • Graphs
      • Just a description of nodes and edges
      • Adjacency Matrix basically states if there is an edge from a to b using matrix[a][b]!=0
      • Adjacency List has a list of all vertices, with a list on each describing destination edges
    • DFS, BFS
      • Depth First Search
        • Uses Stack to develop node path
      • Breadth First Search
        • Uses queue to develop node path
    • Tree Traversals
      • Pre order: data, left, right
      • In order: left, data, right
      • Post order: left, right ,data
  • Smart Pointers
    • Wraps a pointer with a proper destructor
    • unique_ptr
      • only one ptr can point to this object
      • removing or moving that ptr will allow other pointers to point to it
    • shared_ptr
      • multiple ptrs can point to one object
      • can use assigning operators instead of move()
      • maintains a reference counter via use_count()
    • weak_ptr
      • like shared, but no reference counter.
    • make_unique
      • returns a unique pointer of the type.
      • equiv to unique_ptr<T>(new T(args))
    • make_shared
      • same as make_unique, but shared pointer
    • Making these pointers is no different than making a templated class with a pointer as its main variable
  • Design Patterns
    • Generalized solutions and paradigms to programming problems
    • factory
      • Create a polymorphic structure of some class structure
      • Create the factory class which can produce classes from this structure
      • You now ask the factory for a class, and it can return the proper class from the factory
    • iterator
      • Create an iterator and a container that gets an iterator
      • a class that implements container now gets its custom defined iterator
      • that class can be made, with a constructed iterator defined within
    • mediator
      • Create a public class that is implemented within other (tiny) classes to mediate communication
      • i.e. the chat room (mediator) shows messages the users have a function sendMessage() that calls chatroom's showmessage()
    • observer
      • Create classes (observer) that have a reference to other classes (subjects).
      • The subject lets the observers class know when it changes
    • strategy
      • Have a context which references the strategy
      • The strategy has the algorithm we care about
      • The context can cast whatever strategy it needs.
